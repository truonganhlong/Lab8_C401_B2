## 2. Một quyết định kỹ thuật tôi đề xuất

**Quyết định:** Lưu mỗi trace thành **file JSON riêng** (theo `run_id`) thay vì ghi tất cả vào 1 file JSONL duy nhất.

**Vấn đề ban đầu:** Khi implement `eval_trace.py`, cần quyết định cách lưu 15 trace outputs: ghi tất cả vào 1 file JSONL (`artifacts/traces/all_traces.jsonl`) hay tách thành 15 file JSON riêng (`artifacts/traces/run_q01.json`, `run_q02.json`, ...)?

**Phân tích trade-off:**

| Format | Ưu điểm | Nhược điểm |
|--------|---------|-----------|
| 1 file JSONL | Dễ đọc batch, 1 lần `open()` | Không thể tải lại trace đơn lẻ, khó debug từng câu |
| File JSON riêng từng câu | Dễ debug: `cat traces/run_q09.json`, tải lại trace cụ thể để phân tích | Nhiều file hơn, phải `glob()` khi tổng hợp |

**Lý do chọn file JSON riêng:**

Trong quá trình chạy 15 câu, câu `q09` (câu khó nhất, multi-hop) thường cần debug riêng để kiểm tra `workers_called` có ghi đủ 2 workers không. Nếu dùng 1 file JSONL, phải đọc toàn bộ rồi filter — chậm và bất tiện. Với file riêng biệt, tôi chỉ cần mở `run_q09.json` để xem ngay. Ngoài ra, khi pipeline crash ở câu q11, các trace q01–q10 đã ghi vẫn còn nguyên (không mất) vì mỗi câu là 1 file độc lập.

**Bằng chứng từ `eval_trace.py`:**

```python
# Mỗi câu hỏi ghi ra 1 file riêng
run_id = f"run_{q['id']}_{datetime.now().strftime('%Y%m%d_%H%M%S')}"
trace_path = os.path.join("artifacts", "traces", f"{run_id}.json")
with open(trace_path, "w", encoding="utf-8") as f:
    json.dump(trace, f, ensure_ascii=False, indent=2)
print(f"✓ Saved trace: {trace_path}")
```

Kết quả: `analyze_traces()` dùng `glob("artifacts/traces/*.json")` để collect tất cả — sạch và linh hoạt.

## 3. Một lỗi tôi đã sửa

**Lỗi:** `UnicodeDecodeError` khi `analyze_traces()` đọc trace files trên Windows.

**Mô tả lỗi:**

Sau khi chạy `python eval_trace.py` thành công (15/15 traces), bước `analyze_traces()` bị crash:

```
UnicodeDecodeError: 'charmap' codec can't decode byte 0x90 in position 340
File "eval_trace.py", line 189, in analyze_traces
    traces.append(json.load(f))
```

**Nguyên nhân:** Windows default encoding là `cp1252`, nhưng trace files được ghi với `encoding='utf-8'` (có ký tự tiếng Việt như "Phản hồi ban đầu", "Đơn hàng"). Khi `open()` không chỉ định encoding, Python dùng system default — dẫn đến mismatch.

**Cách sửa:**

```diff
- with open(os.path.join(traces_dir, fname)) as f:
+ with open(os.path.join(traces_dir, fname), encoding="utf-8") as f:
    traces.append(json.load(f))
```

**Bằng chứng trước/sau:**

- Trước fix: `python eval_trace.py --analyze` → crash sau khi đọc 1 trace file
- Sau fix: `python eval_trace.py --analyze` → in đủ metrics: `total_traces: 15, avg_confidence: 0.38, avg_latency_ms: 10200`

**Lesson learned:** Trên Windows, luôn chỉ định `encoding='utf-8'` khi đọc/ghi file chứa nội dung tiếng Việt hoặc Unicode.

## 5. Nếu có 2 giờ thêm

**Cải tiến cụ thể:** Thêm LLM intent classifier khi không có keyword match.

**Lý do từ trace:** Trong 15 test questions, `q02` ("Khách hàng có thể yêu cầu hoàn tiền trong bao nhiêu ngày?") được route sang `retrieval_worker` thay vì `policy_tool_worker` vì không có từ khóa "hoàn tiền" hay "refund" trong câu hỏi — chỉ có "hoàn tiền" ẩn trong ngữ nghĩa. Route sai → retrieval lấy sai docs → confidence thấp.

**Thực hiện:** Khi `route_reason` bắt đầu bằng "no keyword matched", thay vì default sang retrieval_worker, gọi thêm:

```python
intent = openai.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": f"Classify: '{task}' → [retrieval, policy, human_review]"}],
    temperature=0
)
```

Chi phí: +~500ms, +~$0.0003/call. Benefit: giảm miss route từ ~13% xuống < 3% dựa trên phân tích trace.
